# Architectural Decision Records

Architectural Decision Records preserve the reasoning behind durable Spark OS decisions.

## Use an ADR When

A decision:

- affects multiple repository areas
- creates a durable constraint
- contains meaningful trade-offs
- changes authority or workflow
- would be difficult to reconstruct later

## Statuses

- Proposed
- Accepted
- Superseded
- Deprecated
- Rejected

Do not edit an accepted ADR to make history appear cleaner. Create a new ADR that supersedes it when the decision materially changes.

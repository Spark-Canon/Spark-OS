# Spark OS Values

**Status:** Framework established — content in development  
**Authority:** Foundational philosophy

## Purpose

Values define what Spark OS refuses to compromise, even when compromise would be faster, easier, cheaper, or more convenient.

A value is not a slogan, preference, target, or temporary priority. It is a durable commitment that should shape the Canon, Academy, governance, operations, AI behavior, and software.

## Relationship to Other Philosophy Documents

- [`manifesto.md`](manifesto.md) explains why Spark OS exists.
- This document defines the commitments that protect that purpose.
- [`principles.md`](principles.md) derives actionable decision rules from accepted values.
- [`mental-models.md`](mental-models.md) helps people reason consistently when applying them.
- [`glossary.md`](glossary.md) defines the language used across all five documents.

Values should constrain Principles. Principles should not invent commitments that have no philosophical basis.

## What Belongs Here

- non-negotiable organizational commitments
- durable beliefs that remain important across changing tools and strategies
- explanations of why each value matters
- observable implications of living the value
- tensions or trade-offs created by the value
- evidence that distinguishes the value from a slogan

## What Does Not Belong Here

- quarterly goals
- performance metrics
- procedures
- software features
- generic virtues with no Spark-specific meaning
- duplicate values that differ only in wording
- principles disguised as values
- aspirations Spark is unwilling to protect through trade-offs

## Value Design Standard

Each accepted value should eventually include:

1. **Name** — memorable and precise
2. **Commitment** — what Spark will protect
3. **Meaning** — what the value means in context
4. **Observable behavior** — what living it looks like
5. **Trade-off** — what Spark may decline or sacrifice to preserve it
6. **Failure mode** — how the value is commonly misunderstood or weakened
7. **Derived principles** — links to rules that operationalize it

## Acceptance Test

Before accepting a proposed value, ask:

- Would Spark keep this commitment when it is inconvenient?
- Does it create meaningful trade-offs?
- Can a decision be challenged by referring to it?
- Is it distinct from the other values?
- Is it durable beyond a specific person, tool, lender, market, or platform?
- Can people recognize whether Spark is living it?

If the answer is no, the statement may be a preference, principle, goal, or slogan rather than a value.

## Future Development

Values should be developed after the Manifesto has established the problem, ambition, and promise of Spark OS.

The initial value set should remain small enough to remember and strong enough to govern difficult decisions.

# Spark OS Principles

**Status:** Framework established — content in development  
**Authority:** Foundational decision rules derived from accepted values

## Purpose

Principles translate Spark OS values into durable rules for making decisions.

They help people act consistently when there is no exact procedure, when multiple options appear reasonable, or when short-term convenience conflicts with long-term integrity.

## Relationship to Other Philosophy Documents

- [`manifesto.md`](manifesto.md) explains the purpose and ambition of Spark OS.
- [`values.md`](values.md) defines what Spark refuses to compromise.
- This document turns those commitments into decision rules.
- [`mental-models.md`](mental-models.md) supplies reusable reasoning tools for applying the principles.
- [`glossary.md`](glossary.md) protects consistent interpretation of important terms.

A principle should be traceable to one or more accepted values.

## What Belongs Here

- durable rules that guide choices across many situations
- decision tests that can be applied before action is taken
- rules for resolving recurring tensions
- statements that connect philosophy to Canon, Academy, governance, operations, AI, and software
- explanations of intended application and known limitations

## What Does Not Belong Here

- step-by-step procedures
- lender-specific rules
- temporary project conventions
- broad aspirations with no decision consequence
- duplicate principles expressed with different wording
- values restated as commands
- exceptions created for one isolated situation

## Principle Design Standard

Each accepted principle should eventually include:

1. **Principle statement** — a concise, actionable rule
2. **Derived from** — the value or values it protects
3. **Rationale** — why the rule matters
4. **Application** — how it changes decisions or behavior
5. **Tension** — what competing concern it may limit
6. **Misuse risk** — how the principle could be overextended or misunderstood
7. **Examples** — representative applications across different parts of Spark OS

## Acceptance Test

Before accepting a proposed principle, ask:

- Does it change how a real decision would be made?
- Is it broader than a procedure but more actionable than a value?
- Can it be applied in more than one domain?
- Is its connection to accepted values clear?
- Does it remain useful when tools, products, and market conditions change?
- Can reasonable people tell when the principle has been violated?

## Future Development

Principles should be drafted only after the initial Values are accepted.

The goal is not to create a large rulebook. The goal is to establish a small set of memorable rules that preserve coherence while allowing professional judgment.

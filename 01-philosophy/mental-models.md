# Spark OS Mental Models

**Status:** Framework established — content in development  
**Authority:** Reusable reasoning frameworks aligned with accepted philosophy

## Purpose

Mental Models define reusable ways of interpreting problems, evaluating trade-offs, and making sense of complex situations.

They support judgment. They do not replace judgment.

A Mental Model should help someone see a problem more clearly, ask better questions, or avoid a recurring reasoning error.

## Relationship to Other Philosophy Documents

- [`manifesto.md`](manifesto.md) establishes why Spark OS exists.
- [`values.md`](values.md) defines what Spark protects.
- [`principles.md`](principles.md) provides durable decision rules.
- This document provides reasoning frameworks for applying those principles in ambiguous situations.
- [`glossary.md`](glossary.md) defines the shared language used to describe them.

Mental Models should reinforce accepted Values and Principles rather than create a competing philosophy.

## What Belongs Here

- reusable frameworks for interpreting recurring problems
- diagnostic questions
- trade-off maps
- patterns that improve professional judgment
- models that connect knowledge, learning, operations, AI, and software
- warnings about predictable reasoning failures
- examples that show where a model is useful and where it is not

## What Does Not Belong Here

- facts that belong in the Canon
- procedures or checklists
- motivational slogans
- untested theories presented as truth
- one-off analogies with no repeatable use
- frameworks that conflict with accepted Values or Principles
- jargon added only to make simple ideas sound sophisticated

## Mental Model Design Standard

Each accepted Mental Model should eventually include:

1. **Name** — concise and memorable
2. **Core idea** — the model in plain language
3. **Useful when** — the situations it helps interpret
4. **Questions to ask** — prompts that activate the model
5. **Failure mode** — what goes wrong when the model is ignored
6. **Limits** — where the model should not be applied
7. **Related principles** — links to the philosophy it supports
8. **Examples** — applications across multiple Spark OS domains

## Acceptance Test

Before accepting a proposed Mental Model, ask:

- Does it improve reasoning rather than merely rename an idea?
- Can it be reused across multiple decisions?
- Does it reveal a trade-off, dependency, pattern, or failure mode?
- Is it understandable without specialized jargon?
- Are its limitations clear?
- Does it support rather than override professional judgment?

## Future Development

Mental Models should emerge from recurring decisions and observed failure patterns.

They should not be invented in bulk. A model becomes durable when repeated use demonstrates that it improves clarity, consistency, or judgment.
